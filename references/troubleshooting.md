| Error Pattern | Symptom | Cause | Solution |
|---------------|---------|-------|----------|
| (Template) | Describe the error message or behavior | Root cause analysis | Step-by-step fix |
| Raw lmms `huggingface` backend fails on local nanoVLM checkpoint | Downstream eval exits with `Can't load image processor ... preprocessor_config.json file` | Local checkpoint folder contains model weights but not the full HF processor bundle expected by `AutoProcessor.from_pretrained` | Keep raw-first/fallback flow: attempt raw backend for compatibility, then fallback to `torchtitan_nanovlm` plugin backend and record backend provenance in `metadata.json` |
| Torchtitan CLI boolean flag mismatch | Training command fails with `Unrecognized options: false` | Torchtitan CLI expects negation flags (for example `--metrics.no-track_peak_memory`) rather than passing string booleans like `false` | Use the correct negation flags (`--*.no-*`) and verify option names with `python -m torchtitan.train --help` before long runs |
| HF streaming fetch transient failure | Run fails during dataset startup with errors like `Temporary failure in name resolution` or `Cannot send a request, as the client has been closed` | Temporary network/DNS instability while reading parquet shards from remote Hugging Face streaming datasets | Re-run after network recovers; avoid treating failed startup runs as parity/performance evidence; keep a successful foreground validation run before launching long tmux jobs |
| Loss or memory regression after moving image flattening into model forward | Step loss drifts from baseline and peak memory rises after collation changes | Changing image container semantics (list-based collation + in-forward flattening) altered runtime behavior and shape flow | Keep images flattened to tensor in dataloader collation for this port, and validate parity with paired 100-step A/B runs |
| Soft-gating loss mismatch when `lr_momh_gate=None` | Torchtitan soft-gating loss diverges early even when speed and VRAM look good | Torchtitan always split `momh_gate` params into their own optimizer group, while baseline keeps them in the language group unless `lr_momh_gate` is explicitly set | In `NanoVLMOptimizersContainer`, only create a dedicated `momh_gate` optimizer group when `lr_momh_gate is not None`; otherwise include gate params in `lm` |
| Baseline finishes all steps but parity harness reports failure | Benchmark exits early with `[parity] baseline run failed` after baseline reaches step target | `nanoVLM_main` can terminate non-cleanly during Python teardown (`PyGILState_Release` / `terminate called without an active exception`) even after training metrics are already complete | In parity harness, treat these teardown signatures as ignorable only when parsed baseline steps already meet target; continue to run Torchtitan and compute paired summary |
| Soft-gating early-step drift despite speed and VRAM gains | Paired 100-step run is faster and lower-memory in Torchtitan but shows large loss spikes around steps `9-14` | Soft-gating numerics are sensitive in early training; whole-window averages can hide instability and model-only compile path can amplify divergence in this port | Enforce an explicit early-step gate on steps `1-15`, compare baseline-vs-baseline and baseline-vs-Torchtitan in the same report, and keep compile scope at model+loss unless model-only proves equal parity and lower VRAM |
| Soft-gating parity regression after removing startup pre-step consume | 20-step parity diverges sharply at startup (for example max diff `0.24875` at step `2`) | Removing the consumed warmup microbatch changes early-step data stream alignment relative to baseline training behavior | Keep startup warmup consume enabled in Torchtitan parity runs; treat no-warmconsume as a known failing variant unless proven otherwise |
| Soft-gating parity regression after iterator/collate validity path rewrites | Mean/max loss diff worsens even though runtime and VRAM remain good | Changes like passing `None` samples through dataset iterator or adding raw-image validity sync alter batch composition/ordering behavior | Keep the proven iterator filtering path and avoid collate/validity rewrites unless a targeted hypothesis demonstrates clear parity improvement |
| Misleading parity conclusions from ultra-short debug runs | 2-5 step A/Bs show loss mismatch that may not reflect long-run parity behavior | Warmup and scheduler semantics differ across stacks when `total_steps` is very small, dominating early loss | For parity claims, align schedule horizon explicitly and include fixed-batch replay + early-window and long-window checks before changing shared runtime |
| Standalone eval fallback is extremely slow (1h+ for partial mmstar) | `torchtitan_nanovlm` fallback keeps running much longer than prior runs despite healthy GPU utilization | Fallback decode path recomputes full sequence (and vision path) every generated token instead of using cached K/V states | Use eval-only KV-cache runtime (`torchtitan/eval/nanovlm_cached_runtime.py`) and shared per-step block-mask reuse; verify with `mmstar` regression that score parity holds and fallback duration returns to practical range |
| Checkpoint key mismatch after refactor | Run fails before step 1 with `Missing key in checkpoint state_dict: dataloader.dp_rank_0.` | Existing `dump_folder` contains checkpoints from a previous trainer/dataloader state schema | Always use a fresh `--dump_folder` for parity A/B after state-related code changes; reserve checkpoint reuse for dedicated compatibility tests |
| Soft-gating FSDP stalls before first step on 2 GPUs | Job reaches `Training starts at step 1` then times out on NCCL collectives | Rank 0 ran a DTensor `full_tensor()` all-gather inside the MoMH post-optimizer metrics hook while other ranks returned early, causing collective-order mismatch | In `torchtitan/models/nanoVLM/hooks.py`, run the hook on all ranks, use `to_local()` for DTensor shards, aggregate stats via a symmetric `all_reduce`, and only publish metrics on rank 0 |
| Vanilla FSDP 2-GPU run times out mid-training with NCCL watchdog | Vanilla run logs step 1 but later aborts with `_ALLGATHER_BASE` / `ALLREDUCE` timeout at `Timeout(ms)=100000` | Environment-specific runtime instability (observed under default `num_workers=2`) during distributed collectives | For benchmark continuity in this environment, rerun with `--dataloader.num_workers 0`; keep this as a known infrastructure/runtime caveat until root cause is isolated |
| 2 GPUs are barely faster than 1 GPU for vanilla training | Multi-GPU wall clock is nearly identical to the 1-GPU control even though distributed setup succeeds | The run is input-bound; built-in TorchTitan timing metrics show data loading consuming roughly `75%` of the step window on `2` GPUs under `num_workers=0` | Check `time_metrics/data_loading(%)` first, treat dataloader optimization as the main speed lever, and do not spend more time on DDP vs FSDP tuning until the input path is improved |
